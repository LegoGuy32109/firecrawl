Non-obvious decisions worth defending

DecisionThe probe to expectThe defense
Sentinel-in-stderr for crossing the Playwright subprocess boundary "Why not return structured JSON?" The replay script runs inside the Playwright service via page.evaluate-shaped exec. The only reliable channel out on failure is the thrown error's message. Appending __FIRECRAWL_REPLAY_FAILURE__:{json} means you keep the human-readable prefix and structured data, parseable from any stderr or error message. See scrape-replay.ts parseReplayFailure().
Screenshot capture is best-effort, never required "What if the screenshot call itself fails?" captureFailureContext() swallows both URL- and screenshot-fetch errors. The envelope ships with whatever was captured. Failing to capture the screenshot must not turn a 422 into a 500.
ZDR strips raw text but keeps structure "Why not also strip actionIndex and selector?" The structure is the bug report; the raw text (pageUrl, screenshot, stderr) is the data. ZDR's contract is "no customer data retained", not "no diagnostics". Privacy projection lives in the responder so this rule is enforced once, not at every call site.
3-second buffer between action-engine timeout and outer scrape timeout in CDP engine "Why the magic number?" If the two timeouts race, the outer abort signal wins and you lose the per-action diagnostic. The buffer guarantees the action engine has time to serialise its SCRAPE_ACTION_ERROR envelope and reply before the outer signal collapses everything to SCRAPE_TIMEOUT. Documented inline at cdp.ts.
Per-token visit counters in the replay-fault site (vs a stateful database) "Why an in-memory Map?" The whole point is reproducible failures with zero setup. Tokens are demo-supplied; counters reset on container restart. Stateful storage would add a dependency you don't need and make demo recovery harder, not easier.
Dummy fc-3d478a296e59403e85c794aba81ffd2a API key prefilled in the playground "Is that a leaked credential?" No. It's a UUID-shaped string the auth middleware accepts because local-db-setup.sql defines a mock auth function that returns success for any non-empty Bearer token. Documented as such in controller.ts with a comment that says "never replace with a live token".
Preact + compat layer, not React "Why not just use React?" Bundle size. Preact + compat is ~3kb gzipped vs React's ~40kb. The playground is served as inline JavaScript from a single HTML response; every byte you ship is a byte the admin page has to parse on every visit. The compat layer means the source still reads like React.
Playground hidden behind {BULL_AUTH_KEY} in the URL path "Why not auth middleware?" It's already behind auth — the bull-board admin pattern. Adding a separate auth layer for an admin-only debugger would mean a second secret to rotate. Reuses the existing admin gate that the team already understands.
E2E tests over unit tests "How do you regression-test this?" Per CLAUDE.md: "E2E (called snips) is always preferred over unit testing." The errors I care about are wire-format guarantees. A unit test that mocks the Playwright service proves nothing useful here; a snip that hits a real fire-engine-or-CDP path proves the entire contract end-to-end. Test gating (TEST_SUITE_SELF_HOSTED, HAS_PLAYWRIGHT, OPENAI_API_KEY) ensures CI runs the right tests on the right matrix.
Error-code regression guard runs in CI "What stops a code from going stale?" The guard test parses every source file's AST, builds a set of every error code referenced from string literals and member-access expressions, and asserts the set is a subset of the enum union. If you rename a code without updating its references, CI catches it. Plain TypeScript misses string-literal references.

Demo choreography

A 10-minute demo that walks all four layers in one run, with recovery for the most likely failure modes.

Cold start to four layers shown on screen in ten minutes. One deliberate failure recovery practised. One backup plan ready if everything is on fire.

Pick a stack — and commit to it now, not Monday

StackUp commandURLPros / cons
Playground compose (recommended for the demo) docker compose -f docker-compose.yaml -f docker-compose.playground.yaml up http://localhost:3002/admin/playground-local/playground Fully containerized; restart: unless-stopped on the api; mounts the Drizzle migration and seed SQL into initdb.d on first volume creation. Slower to start cold (~60-90s) but most resilient mid-demo.
pnpm dev:local (from apps/api/) cd apps/api && pnpm dev:local http://localhost:3002/admin/local/playground One command, fast iteration, you've tuned it heavily. But the harness can be noisy on stderr — Rafa will see tsx reload chatter. Reserve for development; not for screen-share.

If you bring up dev:local for the demo and the playground compose is also up, ports 3001, 3002, 5432 collide. The memory rule: pick one, tear the other down first.

Pre-demo checklist (Monday morning at the latest):

  1. Tear down everything: docker compose down -v from the repo root, then bring up playground compose. This guarantees a fresh volume so the initdb mounts run.
  2. Verify the playground loads at the URL above. Hard-reload (Ctrl-Shift-R) to flush any stale bundle.
  3. Run a smoke scrape against https://example.com. Confirm response shows in the history rail.
  4. Run F5a end-to-end with a throwaway token. Confirm ErrorView shows screenshot + pageUrl + actionIndex.
  5. Record the demo on screen as a backup video. If everything explodes on the call, you fall back to "let me show you the recording while I diagnose."

Probes Rafa will throw mid-demo

What's that ?token= doing?
"The replay-fault server tracks visit counts per token. Fresh token, fresh state. I'm using a timestamped one so the demo doesn't depend on container restart. In-memory Map, no DB — point of the synthetic site is reproducible failure with zero setup."
If Rafa asks "why not a database?", say: stateful storage adds a dependency you don't need; the container restart is faster than seeding anyway.
Is that screenshot full-page or just viewport?
"Viewport, JPEG, quality 70. Full-page would be useful but I went with the cheapest thing that makes the bug obvious. Hot path on every failure, so I capped the cost."
Show you thought about the cost. The screenshot is best-effort and capped — that's a design decision, not laziness.
What if ZDR were on for this team?
"Screenshot is gone, raw URL is gone, but the structure stays — actionIndex, actionType, the code, the catalog explanation and fix. The privacy projection lives in the responder so it's applied to every code path once, not at every call site. Demo: localStorage.setItem('demoZdr', 'true') if you want me to flip it." (don't actually flip it on the call unless you've practised)
Don't actually demo the ZDR mode unless you've drilled it. Easy to forget to flip it back and have it confuse the next demo.
Why is the URL /admin/playground-local/playground and not /admin/playground?
"playground-local is the BULL_AUTH_KEY for the playground compose stack — same pattern as Bull Board uses. It's the existing admin gate. Adding a separate auth layer for an internal debugger would mean a second secret to rotate; this reuses the gate the team already understands."
Naming this without hedging is the move. The BULL_AUTH_KEY pattern is load-bearing.
Can I see the curl version of that?
"Yes — let me pull it up." (Have cli-verification-tests.md open in another tab.) "Same Bearer token, POST /v2/scrape, then POST /v2/scrape/:jobId/interact with the code body. The envelope on the wire is the same as what the Playground renders — Playground is just a UI on top of the same response shape."
Have a terminal pre-warmed with the exact curl. If Rafa asks this and you fumble, the "Playground is just a UI" claim loses force.
Why is this in admin and not a public surface?
"It's a debugger for the team and for me. The public surface is the response envelope itself — that ships to every customer. The Playground is the human view; it doesn't need to be public to do its job. If you wanted it public later, the bundle is one HTML response, no separate static-asset routing — easy to move."
"Easy to move" matters. Rafa might want to know how product-extensible this is. The build script (esbuild → single bundle, inlined in controller.ts) is intentionally portable.

Recovery playbook — what to do when something breaks live

SymptomLikely cause30-second fixWhat to say
Replay-fault scrape returns the no-button page on visit 1 Token reused from your pre-demo testing. New token: append -2 or timestamp. "Let me grab a fresh token — these have visit counters." (don't apologise; this is by design)
Playground returns 404 or "Playground bundle not found" Bundle wasn't built. pnpm build:playground didn't run. In another terminal: cd apps/api && pnpm build:playground, then hard-reload. "Bundle's stale — building." Then continue.
API returns auth function pgrst_anon_role does not exist or similar Postgres volume was reused without the seed running (initdb only fires on first creation). docker exec -i firecrawl-nuq-postgres-1 psql -U postgres < apps/api/scripts/local-db-setup.sql "DB volume's stale, applying seed." (per memory `feedback-dev-workflow`)
Interact returns JOB_NOT_FOUND USE_DB_AUTHENTICATION isn't true, so log_job skipped the DB write. Or you're on the wrong stack. Confirm playground compose is the active stack; if dev:local, restart with the env var set. "Persistence is off — switching stacks." (rare on playground compose; common on dev:local)
Connection refused on replay-fault:4322 Container exited or never started. docker compose ps to check; docker compose up -d replay-fault to restart. "replay-fault's down, kicking it." It's a 100-line node server; it doesn't fail often.
Everything is on fire and Rafa is watching Don't diagnose live. "Let me show you the recording while I diagnose in the background." Have the pre-recorded demo open in a tab. Composure beats correctness here. Acknowledge the bug, show the recording, debug after the call.

The four-layer thesis

Once Rafa is past "why this customer", he wants to know why this much. Drill the four-layer model.

You can recite the four layers in order, name the organising principle, and defend each layer against the cheapest alternative — all without hedging.

The four layers (memorise this)

#LayerThe jobThe cheap alternative you rejected
1Typed error & warning catalogMake it impossible to ship a failure without recording where it happened.Better strings inside the existing free-form error field.
2Response envelope (the responder)One place that owns diagnostics, privacy projection, and step-level traces — so the rule is enforced, not begged for.A handful of helper functions every controller calls.
3Local playwright;cdp engineA laptop-sized lab to exercise the contract end-to-end without burning fire-engine credits.Iterate against cloud fire-engine.
4Admin PlaygroundThe surface a human (you, Rafa, on-call) actually looks at to verify the contract works.Print JSON to a terminal during the demo.

The organising principle: each layer is the cheapest possible thing that forces the next one to behave. The catalog forces engines to capture context; the responder forces routes to project it; the CDP engine lets you iterate; the Playground lets you see it.

The thesis drill

Walk me through the four layers in order.
"Bottom up: a typed error catalog where every code carries an httpStatus, an explanation, a fix, and a typed details shape. A response enveloper, makeResponder(req, res), that every v2 route resolves through — it owns diagnostics, applies privacy, emits step-level traces. A local CDP engine, playwright;cdp, registered alongside the existing one, so I could exercise the contract on my laptop. And an admin Playground that renders the envelope: ErrorView, DiagnosticsWaterfall, the request builder."
Bottom-up order matters: it shows the layers depend on each other. Top-down would imply the Playground was the goal, which would weaken the catalog argument.
Why four layers? You could have built just the error envelope and called it done.
"Each layer fixes a problem the others don't. The catalog without the responder is a type system nobody enforces — controllers still throw whatever they want. The responder without the catalog has nothing to project. The CDP engine without either is just an engine; the contracts are what make it a lab. The Playground without the rest is a JSON viewer. Cut any one and the argument doesn't hold."
This is the load-bearing answer. Practise it. The structure is "X without Y is ___" four times — once for each layer being the missing one. Don't get cute with the symmetry; let it speak.
If you had to ship only one of the four, which?
"The responder, layer 2. It's the place the privacy rule and the diagnostics contract get enforced — losing it means every controller has to remember to do those things, and they won't. The catalog is the schema, the engine is the lab, the Playground is the surface — but the responder is the load-bearing piece."
Don't pick the catalog, even though it's the headline. The catalog is data; the responder is the control point. Picking the responder shows you know what actually enforces the contract.
If they share an organising principle, name it in one sentence.
"Each layer is the cheapest thing that forces the next layer to behave — the type system forces engines to record context, the responder forces routes to project it, the CDP engine forces the loop to be fast, the Playground forces somebody to actually look at the output."
"Forces" is the word doing the work. It separates this build from a stack of helpful utilities. Engineers like utilities; product engineers ship constraints.

One probe per layer (this is where Rafa will spend most of the time)

[Layer 1] Why a typed catalog and not just enum codes?
"Codes alone don't tell the SDK or the operator what to do with them. The catalog pairs every code with an httpStatus, an explanation, and a fix. httpStatus means the responder doesn't need a switch statement — code in, status out. explanation and fix mean the customer sees the same words whether they're reading the API response, the docs, or the SDK exception. The typed details shape — for SCRAPE_ACTION_ERROR that's { actionIndex, selector, pageUrl, screenshot, actionType, actionStatuses[] } — means the engine code can't ship a failure without recording where it happened. The type checker rejects it."
The "type checker rejects it" is the key line. Frame the catalog as a constraint, not a convenience.
[Layer 2] Why a responder, not just helpers?
"Helpers are opt-in; a responder is the one channel a route can use. makeResponder(req, res): every v2 route now resolves through it. It owns the diagnostics struct, applies privacy mode (ZDR strips raw text but keeps structure), and emits step-level traces via responder.step({...}). A 14-step run produces 14 diagnostic entries instead of one boolean. The privacy rule lives in one place — not in forty controllers."
If Rafa says "but you could have just made the helper required" — agree, then say: "a helper you have to remember to call is a helper you forget to call. The responder is unforgettable because it's the route's response object."
[Layer 3] You added an entire new engine. Why not just use fire-engine?
"Iteration speed. Fire-engine is shared, costs credits, has round-trip latency. To prove the catalog and the responder worked, I needed to trigger SCRAPE_ACTION_ERROR on a 14-step replay, watch the diagnostics waterfall populate, fix the projection, and re-run — fifty times, in an afternoon. Doing that against fire-engine would have burned credits and put me in a shared sandbox with the rest of the team. The CDP engine connects to a local Playwright service over the Chrome DevTools Protocol and exposes the full feature surface the failure path needs — actions, screenshots, mobile emulation, geolocation. The old playwright engine is untouched; it's still there as a fallback for callers who don't need any of that."
If Rafa pushes — "isn't this duplicating production?" — say: "production stays on fire-engine. The engine picker prefers the higher-quality engine when its features match. There's no regression for anyone who didn't opt in." Then point at engines/index.ts.
[Layer 4] You spent time on a UI. Why?
"Because the contract only matters if somebody looks at it. The Playground is the surface where the diagnostics get rendered — ErrorView leads with 'Action 3 failed: #nav-login', the screenshot, and the catalog's explanation and fix. DiagnosticsWaterfall renders the per-step trace as a timeline. The point isn't the UI; the point is that an internal tool that's used means the contract gets tested. It's also the surface I'd defend the prototype with — if I'd just printed JSON to a terminal you and I would be reading curl output right now, and the failure frame wouldn't read like a bug report."
"An internal tool that's used means the contract gets tested" — that's the line. Dogfooding is one of the role posting's stated themes. Drop it here.

The flow — what happens when step 11 of 14 fails

If Rafa asks "trace it through," he's testing whether you actually know how the layers interact at runtime. Have this answer pre-cached.

Walk me through what happens when step 11 of a 14-step interact replay fails on a selector miss.
"The replay script is built by buildReplayScript() in scrape-replay.ts. Inside the Playwright service, the action loop hits step 11, page.click(selector) throws. The failReplay() helper catches it, builds a structured JSON payload {actionIndex, actionNumber, actionType, reason}, appends it to the error message after the sentinel __FIRECRAWL_REPLAY_FAILURE__, and re-throws. The Playwright service returns a 422 with that message in stderr. Back in controllers/v2/scrape-browser.ts, parseBrowserExecFailure() reads the body, parseReplayFailure() finds the sentinel and parses the JSON. Meanwhile captureFailureContext() hits the browser service twice — once for the live URL, once for a screenshot — both best-effort, both swallowed on error. The responder gets BrowserError.EXECUTION_FAILED with the typed details — replayFailedAt, pageUrl, screenshot, stderrSnippet — and writes it through the privacy projection. The customer sees the structured envelope; the Playground renders ErrorView; the DiagnosticsWaterfall shows ten green rows and one red."
This is the hardest answer to deliver smoothly. Practise it. The names (parseReplayFailure, captureFailureContext) signal you know your own code. Don't paraphrase them.

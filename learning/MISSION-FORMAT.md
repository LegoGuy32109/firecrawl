# MISSION-FORMAT

A mission document is a short statement of *why the user is learning this topic at this point in time*. Every lesson, reference, and learning record in this workspace should be traceable back to it.

A good mission document has:

1. **A single concrete goal**, ideally tied to a date or stakeholder. Not "get better at Firecrawl" — "defend commit X to team Y on date Z."
2. **A framing of the call/test/event** the user is preparing for. What is the shape of success?
3. **Why this is the mission and not a broader/narrower one.** What is the user *not* trying to learn?
4. **Win conditions** — observable behaviors that would prove the mission is accomplished.
5. **Out of scope** — what is deliberately not being learned, so lessons don't drift.
6. **Pillars** — 2-4 thematic categories that lessons fall into. These should be derived from the user's actual prep needs, not from a generic taxonomy.

Update the mission (and add a learning record) whenever the user's goal changes meaningfully — e.g., the call is rescheduled, the scope shifts from "defend" to "extend," or a new stakeholder enters the picture.
